### Coupling concerns

#### F54-A01 — per-source CSVs serve dual purpose (cache + authoritative) — LOW (architectural)

Same as F54-01 from a different lens. The repository treats
per-source CSVs as both:

- **Cache**: regenerated by `fetch_source`, can be trusted only as
  long as sensors.json hasn't drifted.
- **Authoritative store**: `_load_per_source_csvs` reads
  matched_sensors verbatim and feeds it into the merge.

The cleanest fix is to declare per-source CSVs as caches:
- On load, drop matched_sensors and let merge/derive recompute, OR
- On load, call `derive_spec` to refresh.

The alternative — treating CSVs as authoritative — would require
a separate "user-overrides" file for sensor name updates and is
overkill for a static-site generator.
